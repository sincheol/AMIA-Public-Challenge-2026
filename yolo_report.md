# AMIA 2026: YOLO 모델링 파이프라인 분석 리포트

## 1. 개요 (Overview)
본 리포트는 `amia-public-challenge-2026` 흉부 X-ray 객체 탐지 대회에서 YOLO(You Only Look Once) 프레임워크를 기반으로 수행한 과정(데이터 분석 → 전처리 → 모델 학습 → 튜닝 → 캐글 제출)을 정리한 문서입니다.

초기 EDA부터 시작해, 모델이 0.05점대에서 0.428점을 달성하기까지 적용한 트러블슈팅 및 베이스라인 모델 구축(`model_yolo_v1.py`) 내역을 단계별로 기록했습니다.

---

## 2. 단계별 분석 (Deep Dive by Steps)

### Step 1. 탐색적 데이터 분석 (EDA) 및 대응 전략
`eda_step1~3.py`를 통해 데이터를 분석(`eda_review_report.md`)한 결과, X-ray 데이터셋의 4가지 주요 특성을 식별하고 모델링 전략에 반영했습니다.

1. **클래스 불균형 (Class Imbalance)**: `Pneumothorax`(195개)와 `Aortic enlargement`(5,481개) 간의 빈도 차이가 컸습니다. 
   - **전략**: 별도의 Focal Loss 파라미터 조작 대신, YOLOv8의 모자이크(`mosaic=1.0`)와 믹스업(`mixup=0.1`) 증강 기법을 사용하고 분류 손실 가중치(`cls_pw=1.0`)를 조절하여 불균형 완화를 시도했습니다.
2. **높은 정상 이미지 비율**: 전체 이미지의 약 60.4%(5,175장)가 질환이 없는 정상(`No finding`) 이미지로 구성되었습니다.
   - **전략**: 정상 이미지를 제외하지 않고, 오탐(False Positive)을 억제하기 위한 Background 이미지로 활용했습니다.
3. **고밀도 다중 병변 (Density)**: 한 이미지에 최대 57개의 질환 바운딩 박스가 겹쳐서 나타납니다.
   - **전략**: 추론 시 최대 탐지 개수 상한을 `max_det=300`으로 늘려 병변 누락을 방지했습니다.
4. **다양한 병변 크기**: 결절(Nodule)부터 기흉(Pneumothorax)까지 스펙트럼이 넓어 다중 스케일 탐지가 가능한 YOLOv8 아키텍처(FPN/PANet 기반)를 채택했습니다.

---

### Step 2. 통합 전처리 및 데이터 포맷팅
Kaggle 환경과 YOLO의 학습 포맷에 맞추기 위해 전처리 파이프라인을 구축했습니다.

1. **동적 파일 확장자 매칭**: 단일 확장자 하드코딩 대신 `(".png", ".jpg", ".dcm", ...)` 튜플을 사용한 재귀적 탐색(`os.listdir`)을 구현하여 확장자 변동에 대응했습니다.
2. **배경(Background) 처리 로직**: 정상(`No finding`) 이미지의 경우 훈련 폴더에 빈 `.txt` 파일(`touch {image_id}.txt`)을 생성하여, YOLO가 해당 이미지를 배경으로 인식하도록 처리했습니다.
3. **YOLO 정규화 좌표 변환**: 원본 픽셀 좌표(`x_min, y_min, x_max, y_max`)를 YOLO 포맷인 중심 좌표 및 너비/높이 상대 비율(`class_id, x_center, y_center, width, height`)로 변환하여 텍스트 파일로 저장했습니다.

---

### Step 3. 초기 베이스라인 모델 구축
본 모델링 코드는 고도화 튜닝(Step 6)을 진행하기 전, 안정적인 학습 파이프라인을 구축하기 위한 초기 베이스라인(v1)입니다.

#### 1) 시스템 및 환경 설정
- **GPU SM capability 자동 감지**: 할당된 GPU 아키텍처(P100, T4 등)를 확인하여 알맞은 버전의 PyTorch를 설치하는 분기 로직을 구현했습니다. 이를 통해 `cudaErrorNoKernelImageForDevice` 에러를 방지했습니다.
- **메모리 단편화 OOM 방지**: 2에폭 부근에서 발생하던 메모리 단편화(Fragmentation)로 인한 OOM 현상을 완화하기 위해 `PYTORCH_ALLOC_CONF=expandable_segments:True` 환경변수를 설정했습니다.
- **Kaggle 데이터 경로 탐색**: `/kaggle/input/competitions/...` 와 같이 경로가 달라질 수 있는 캐글 환경을 고려해 2단계 하위 폴더까지 탐색하는 `_find_kaggle_input()` 함수를 추가했습니다.
- **캐시(Cache) 비활성화**: RAM(26GB)과 Disk(19GB) 용량이 캐시 요구량(약 30GB)에 미달하여 `CACHE=False`로 비활성화했습니다.

#### 2) DDP(분산 처리) 이슈 패치
- 2개의 T4 GPU에서 `device=[0, 1]`로 DDP 학습 시, 정상(Background-only) 배치에서 DFL 파라미터가 Gradient를 받지 못해 에러가 발생했습니다.
- **해결**: 런타임에 Ultralytics 소스 코드를 수정(`_patch_ultralytics_ddp()`)하여 `find_unused_parameters=True`가 적용되도록 패치했습니다.

#### 3) 평가 지표(Metric) 별도 측정
- 대회 공식 지표인 **mAP@IoU=0.4** 측정을 위해 `torchmetrics.MeanAveragePrecision`을 도입하여 채점 루프를 구현했습니다.

#### 4) 모델 및 파라미터 세팅
- **VRAM 고려 설정**: 15GB VRAM 환경에 맞춰 `yolov8m.pt` (Medium) 모델을 사용하고, 해상도 `1024x1024`, 배치 사이즈 `BATCH=28`로 설정했습니다.
- **데이터 증강**: 일반화 성능을 높이기 위해 `mosaic=1.0`, `mixup=0.1`과 함께 **`cutmix=0.1`** 파라미터를 추가 적용했습니다.

---

### Step 4. 추론 최적화 및 후처리 (Kaggle Submission)
제출 점수가 0.055점으로 낮게 나오던 현상을 개선하기 위해 적용한 트러블슈팅 내역입니다.

1. **원본 해상도 역스케일링 (Rescale)**
   - 모델은 `1024x1024` 리사이즈 이미지로 훈련했으나, 캐글 채점 서버는 `img_size.csv`의 원본 해상도를 기준으로 채점합니다.
   - **해결**: 추론 결과(`r.orig_shape`)에서 이미지 너비 변수(`orig_w`)를 가져와 `scale_x = true_w / orig_w` 로 비율을 동적으로 계산하여 예측 좌표에 곱해주는 스케일링 로직을 구현했습니다.
2. **임계값(Conf=0.01) 하향 조정**
   - 질환 박스가 컷오프되어 재현율(Recall)이 낮아지는 것을 막기 위해 추론 임계값(`INFER_CONF`)을 `0.01`로 설정했습니다.
3. **정상(No finding) 폴백(Fallback) 로직**
   - 낮은 임계값 설정으로 정상 이미지에도 노이즈 박스가 다수 발생하는 부작용이 있었습니다.
   - **해결**: 이미지 내 예측 박스 중 "최고 신뢰도가 0.15 이하"인 경우 예측 박스를 모두 제거하고 `14 1.0 0 0 1 1` (정상 라벨)로 제출하도록 예외 처리했습니다. 이를 통해 정상 이미지의 오탐을 줄였습니다.

---

## 3. 결과 및 요약 (Conclusion)
- **최종 Public Score**: `0.052` → `0.4100` → **`0.428`** 
- **결과 요약**: 
  - 해상도 스케일링 방식을 수정하여 `0.4100`점을 확보했으며, 이후 **정상 클래스 폴백 로직**과 **저임계값 탐색**을 조합하여 0.428점으로 점수를 상승시켰습니다.
- **의의**: 데이터 특성을 반영한 전처리와 인프라/프레임워크 환경 최적화(DDP 패치, 메모리 관리)를 통해 안정적인 V1 베이스라인 파이프라인을 구축했습니다.
